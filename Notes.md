Findings

Business detail hydration can keep stale data alive across navigations instead of accepting newer server data. In useBusinessDetailsController.ts (line 40) the store is only refreshed when shouldSyncBusinessDetailsEntry(...) returns true, but business-details-state.ts (line 121) only compares business.updated_at, financeDashboard.generatedAt, and array lengths. Then resolveBusinessDetailsView(...) (line 141) prefers existing store arrays over the fresh server payload. If inventory quantities, employee assignments, shelf contents, or upgrades change without changing array length or business.updated_at, revisiting the page can render stale data indefinitely.

Optimistic rollback is slice-wide and can erase unrelated realtime updates that arrive while a mutation is in flight. runOptimisticUpdate (line 37) snapshots the whole slice before the optimistic write, then restores that old snapshot on failure. Any concurrent realtime patch to the same slice is lost. That is especially risky in high-churn slices like market state, where mutations are frequent and realtime refreshes are active in realtime-provider.tsx (line 427). This will produce hard-to-reason state regressions after transient API failures.

Market optimistic writes also rebuild arrays from render-time closures, which can overwrite fresher store state even on success. In MarketClient.tsx (line 360) and MarketClient.tsx (line 430), patchMarket is fed arrays derived from captured listings/transactions values, not from the latest store state. If a realtime event lands between render and mutation completion, those writes can drop new listings, quantity changes, or transactions. This is the same class of stale-state drift as above, but happening even without rollback.

Data assembly is duplicated in too many places, which is already drifting architecturally and will make updates expensive. The same business-detail payload is built separately in the page loader at [id]/page.tsx (line 41), the API route at state/route.ts (line 39), and the realtime refresh path at realtime-provider.tsx (line 191). The broader page loaders are also duplicated between server loaders in server-data.ts (line 105) and client fetchers in queries.ts (line 164). This is a maintainability issue now, not just style: any schema or business-rule change has to be updated in multiple places.

React Query is currently dead infrastructure. The provider and client are wired in AuthenticatedQueryProvider.tsx (line 12) and query-client.ts (line 3), but the codebase has no useQuery, useMutation, hydration boundary, or cache invalidation usage. Right now the app is paying the complexity cost of two client-state strategies while only using Zustand. Either lean into React Query for fetch/cache synchronization or remove it.
